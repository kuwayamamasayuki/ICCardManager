# docs ↔ 実装 ドリフト監査 ワークフロー

docs を「正典(こうあるべき)」、実装コードを「真実(実際にこう動く)」とし、両方向のズレを優先度付きレポートにまとめる。

**この監査は Claude の dynamic Workflow(Workflow ツール)で、ultracode 前提で走らせる。** メイン(司令)が plain JavaScript のオーケストレータを 1 本書いて Workflow ツールで起動し、その中で reader / verifier を `agent()` として `parallel()`(同じ round の中で同時起動してバリア同期)/ `pipeline()`(段をまたいでつなぐ)で並列に回す。ループ・分岐は JS で書く。「サブエージェントを spawn」「可能なら Workflow、使えなければ Task / Agent」のような両論併記はしない ── 実行基盤は Workflow ツール一本。

**1 コンテキストで読んで「ドリフトだ」と 1 人で断じるのは禁止。** 読み手(reader)と疑い手(verifier)は必ず別 `agent()` にする。これがこのワークフローの肝で、単一エージェントの確信は敵対的検証を通すまで採用しない。エージェントの「緑」(検証 OK の自己申告)も鵜呑みにせず、最終判定はメインが実ファイルを読んで確定する。

ultracode は opt-in(キーワードまたはセッションで ON)で、Workflow を起動する前提条件である。**ultracode が OFF のときは、(1) ユーザーに ON にしてもらってから本ワークフローを回すか、(2) どうしても今すぐなら『さっと 1 round』(主要 doc とコードだけを 1 巡読んで明白なズレだけ拾い、敵対的検証なし)に落として、その旨を明記する ── のどちらかにする。** 深く回したつもりで実は単一コンテキストで読んでいた、を起こさない。

このワークフローは特定のリポジトリ事実を仮定しない。固有の事実はまず §1 で発見・確認する。記憶や「たぶんこうだろう」で埋めない。起動引数があれば監査範囲のヒントに使う(例「docs/ と app/src/」「ランキング周りだけ」)。無ければ §0 で確認するか全体を対象にする。

このリポジトリの既存規律(CLAUDE.md / AGENTS.md)に従う: 自分が作った PR は CI 緑なら自動 squash merge してよいが、検証の緑は自分の目で読んでから(gate-on-green)。docs は最新仕様だけ・現在形で書き履歴を足さない。作業は worktree 隔離で。途中の各論では止まらず根拠を持って前進し、確認するのは §0 のモード選択 / 不可逆操作(merge / push / deploy / 削除) / §6 の安全策に抵触する判断、の 3 つだけ。

## 0. 最初に確認する(ループ条件・モード・範囲。確認できたら自走)

着手前に次の **3 点を 1 度にまとめて確認する**(細切れに聞かない)。

**「深さ」は別個に聞かない ── 深さ = ultracode** だからである。ultracode が ON なら、それ自体が「Workflow で fan-out して敵対的検証まで厚く回す深い監査をする」という宣言になる。深さの代わりに、実際に効くノブである **ループの止めどき**を最初に決める。

1. **フィードバックループの条件(回数 / 停止条件 / 上限)** ← 一番大事:
   - **既定(推奨)**: 新規 finding(これまでの round で出ていないドリフト)が **連続 2 round ゼロ**になったら停止(loop-until-dry)/ **最大 5 round で打ち切り** / 敵対的検証は重大度 **high 以上を独立 2 名以上で多数決**(過半数が「本物でない」なら捨てる)。
   - **さっと**: **1 round** で締める(fan-out 1 巡 + 軽い検証のみ、多数決はスキップ)。ざっと当たりを付けたいとき。
   - 数字はユーザーが上書きできる(例「連続 3 round ゼロ / 最大 8 round」「high 以上は 3 名で」)。上書きが無ければ既定で回す。
2. **動作モード**: A) **レポートのみ(既定・推奨。** コードも doc も変更せず、全エージェントを読み取り専用で動かす) / B) 修正まで(レポート提示 → 承認後、「安全・明白」な修正だけを小さな単一目的 PR で。判断物は人間に残す)。明示が無ければ A。
3. **監査範囲と既知の意図**: 重点 / 除外対象(自動生成物・seed・生成された型等)、わざとそうしている既知の例外。無ければ全体対象で自動生成物・`node_modules`・`dist` 等を除外。

ultracode が OFF なら、3 点を聞くより先に「ON にして深く回すか / さっと 1 round に落とすか」を確認する(冒頭参照)。回答待ちの間に §1 の発見は並行で進めてよい(読み取りのみ)。

## 1. リポジトリ・プロファイルの発見(推測禁止)

実ファイル・実コマンドで地図を確定し、全サブエージェントに共通コンテキストとして渡す。

- **正典の序列**: `CLAUDE.md` / `AGENTS.md` / `README*` / `docs/`(このリポジトリは docs/ が仕様の正典)/ `.agents/skills/` / `.claude/commands/` / `.github/` / migration・schema 等「仕様としての設定ファイル」。矛盾時にどれが勝つか(philosophy / docs > CLAUDE.md 等)の宣言を探し、無ければ「正典の序列が未定義」を記録する。
- **検証コマンド**: `package.json` scripts(`npm run typecheck` / `lint` / `test`)と作業ディレクトリ(app/ 等)を実在確認する。CI(`.github/workflows/`)が回すコマンドを正典とする。副作用(deploy / 外部送信 / DB 変更)を持つコマンドは実行前に確認し、あれば実行せず「要人間判断」へ。
- **PR・ブランチ・コミット規約**: 既定ブランチ、ブランチ命名、Conventional Commits、`gh` 利用、CI gate、レビュー / マージ規約、コミットの言語・末尾署名。`CLAUDE.md` / `AGENTS.md` / `.github/` / `git log` を一次情報として読む。
- **作業隔離**: worktree を最初の一手で切る(リポジトリ階層内、一時ディレクトリ不可)。読み取り専用でも現在の作業ツリーの状態を変える操作はしない。
- **機械ガード**: 既存の lint ルール / カスタムテスト(「この語を使うな」等の制約をコード化したもの)を把握する(ルール変更時に同じ場所を更新するため)。

確定できない項目は「未確定」と明記する(推測で埋めない)。

## 2. ドリフト 9 種別(各 finding に 1 タグ)と重大度

- **D1 願望/陳腐化 doc**: doc にあるがコードに無い / 名前違い / 挙動違い。doc・README 内のコード例・コマンド例の腐りも含む。
- **D2 取り残された doc**: コードはやっているが doc に無い / 否定している。安全・プライバシー・法務に直結しやすく重大度が上がる(このリポジトリなら L2 不変条件・個人ログをサーバに置かない・anti-food-log 等に関わるズレ)。
- **D3 履歴・経緯 cruft**: 「以前は」「Phase」「PR#」「○月に撤回」等。git log / CHANGELOG / roadmap へ分離。鮮度マーカー(「○年○月時点の最新」)は残す。
- **D4 陳腐化した制約**: 「できない/禁止」を (a) 恒久理由(L2 / ToS / anti-food-log / 体験原則 / 安全 / コスト / 設計の芯)=守る、(b) 当時の技術前提=最新を一次確認して更新候補、に分類。見極め不能は守る側。一次確認なしに消さない。
- **D5 正典の分裂・重複・矛盾**: 同じ事実が複数 doc に重複・矛盾。正典を 1 つに定め他は参照に。
- **D6 壊れた参照**: 死んだリンク/アンカー、削除済み要素・スキル・コマンドへの言及、移動パスの旧参照、デッドリンク。
- **D7 デッドコード・残骸**: 未参照の関数/フラグ/設定、到達不能分岐、固定化済みなのに残る分岐。動的参照・文字列経由・テスト専用・export・外部インポートを潰してから断定。
- **D8 inert な「実装済み」(見落としの筆頭)**: 部品はあるが配線・到達経路が無く動かない。doc が約束する機能ごとに入口→ハンドラ→永続化→集計→表示の経路を 1 本辿り、途切れを探す。「定義がある」と「呼ばれている」を区別。
- **D9 用語・命名のゆれ**: 利用者向け語彙とコード内部語の取り違え、同義語の乱立、表記ゆれ(語彙正典があれば照合)。ただし利用者向けコピーと内部スキーマ名を一括置換で混ぜない。

重大度: critical(安全/プライバシー/法務/データ消失/課金に直結) / high(doc を信じると確実に詰まる、inert な実装済み、壊れた手順) / medium(誤解を生むが回避可) / low(整頓) / info(observation 枠、断定しない)。各自の感覚でなくこの基準で付ける。

## 3. 段取り(Workflow ツールで fan-out → 敵対検証 → 統合 を loop-until-dry)

**Workflow ツールで回す。** メインが plain JavaScript のオーケストレータを 1 本書き、その中で reader / verifier を `agent()` として起動する(1 コンテキストで役割を演じない。構造化出力はプレーンオブジェクトの schema を `agent(prompt, { schema })` で受ける)。1 round の中身は **reader を `parallel()` で並列起動 → 上がってきた finding を別 `agent()`(verifier)に渡して敵対的に検証 → confirmed / needs_human を統合**、の 3 段(段をまたいでつなぐときは `pipeline()`)。これを **新規 finding が連続 2 round ゼロになるまで繰り返す(loop-until-dry、最大 5 round)**。round をまたいで既出 finding は seen set で dedup し、新規だけを次 round の対象にする。規模目安: 1 round の reader は領域数(小 3〜5 / 中 6〜10 / 大はテーマごとに二段 fan-out)。

1. **フェーズ A 並列読解**: doc とコードを領域分割し(例: 原理・哲学 / データ・スキーマ / ランキング / 候補抽出・外部API / 認証・権限 / プライバシー(L2)/ ビルド・配信 / 規約・ポリシー / UI・利用者向けコピー / 設定・フラグ / 用語)、領域ごとに reader を並列起動。各 reader に §1 プロファイル + §2 種別を渡す。doc 主張↔コード実挙動を双方向に突き合わせ、D8 のため機能ごとに呼び出し経路を端から端まで辿る。各 finding は `id / drift_type / claim / doc_evidence(path:line) / code_evidence(path:line) / expected / actual / severity / proposed_fix(既定 fix_doc) / confidence / notes` で返す。**file:line 無き主張は出さない。秘密情報の値は転記せず場所だけ参照しマスク。**
2. **フェーズ B 敵対的検証**: 起票していない別エージェントが実ファイルを読み直し、引用の正確さ・文脈・条件分岐・別 doc での撤回/補足・動的参照・(D4 は)一次情報を潰して `confirmed / refuted / needs_human` を裁定。high 以上は独立 2 名 + 多数決、割れたら needs_human。確信なければ refuted か needs_human に倒す。reader の高 confidence より検証担当の refute を優先。副作用の無い検証コマンドだけ実際に走らせる(記憶で裁定しない)。
3. **フェーズ C 統合**: confirmed と needs_human を dedup・マージし、優先度(正しさ・安全 > 誤解を生む不整合 > 整頓)で並べる。refuted は件数と一行サマリだけ残す。

## 4. レポート(両モード共通の成果物)

チャットに提示する(新たな履歴 doc として恒久コミットしない、秘密情報の値は載せない):

1. リポジトリ・プロファイル(§1。未確定は明記)
2. サマリ(重大度 × 種別の件数、最も効く 3〜5 件、全体傾向、refuted 件数)
3. 確証ありの所見一覧(優先度順、各件に「安全・明白に自動修正可」/「要人間判断」タグ)
4. 要人間判断リスト(doc/コードどちらが正か曖昧・制約緩和(D4)・実装側修正・正典序列未定義。論点+選択肢+推奨)
5. 検討したが本物でなかった一覧(refuted、一行ずつ)
6. (モード B 向け)PR 分割案(1 PR = 1 論理変更)

## 5. モード B — 承認後の修正

ユーザーがレポートを承認したら、「安全・明白に自動修正可」タグの付いた発見だけを修正する。**「安全・明白」= ①修正方針が一意 ②doc 側のみで実装挙動を変えない ③制約緩和/正典序列変更を含まない ④検証済み**(典型: デッドリンク、削除済み要素への参照、明白な誤記・古いパス・古いコマンド、cruft 除去、一意な用語ゆれ統一)。これ以外は needs_human に残す。

- 既定は「doc を実態に合わせて直す」。実態がバグなら実装修正(ただし needs_human 扱いで慎重に)。曖昧は直さず needs_human。
- doc は最新仕様だけ・現在形(履歴を足さない、最新仕様と同じ結論を各所に複製しない)。経緯は git log、計画は roadmap、変更履歴は CHANGELOG。鮮度マーカーは例外。
- 陳腐化した制約を現在形に更新するときは最新を一次確認してから。フォールバック/但し書きで安全と最新化を両立できるなら添える。
- ルール・制約を変えたら機械ガード(テスト/lint)を同じ PR で更新する。
- 1 PR = 1 論理変更。doc 修正と code 修正を混ぜない。指摘範囲を勝手に広げない。「ついでに」リファクタを混ぜない。頼まれていない新ルール・新フラグを発明しない。
- 検証ゲート: PR を出す前に typecheck / lint / test を自分で走らせ、緑を自分の目で読んでから完了と言う。サブエージェントの「緑」を鵜呑みにしない。失敗テストの期待値を書き換えてバグを隠さない。検証と公開(commit/push/PR)を同じバッチに混ぜない。
- 不可逆操作はサブエージェントにさせない。commit / push / PR はメインが行い、merge は CI 緑を確認のうえメインが行う(破壊的 SQL / データ消失 / 金額発生 / L2・ToS に関わるものだけ人間承認)。

## 6. 全エージェント共通の安全策(これを破る進め方・主張・修正は失敗)

1. 証拠(正確な file:line と「期待 vs 実際」)なき断定をしない / 捏造しない。存在を確認できないファイル・関数・コマンド・ドメインを修正で新たに書き込まない。
2. 一次情報で裏を取る。実ファイルを読む。技術的事実(SDK・ライブラリ・外部サービスの現在仕様)は公式 docs・レジストリ・実行で確認する。「記憶では/以前は/たぶん」が頭をよぎったら書く前に一次情報を引く。
3. 単一エージェントの「ドリフトだ」を検証なしに採用しない。検証を通らないものは needs_human / refuted に分ける。
4. 既定は「doc を実態に合わせて直す」。実態がバグなら実装を直す。曖昧なら needs_human。
5. 「禁止 / できない」を消す前に理由を見極める((a)恒久=守る /(b)技術前提=最新確認して更新)。見極め不能は守る側。惰性で守るのも雰囲気で消すのも両方禁止。
6. doc は最新仕様だけ・現在形。履歴・経緯・事故記録は git log / CHANGELOG / roadmap へ。最新仕様と同じ結論を各所に書き足して膨らませない。鮮度マーカーは例外。
7. ルール・制約を変えたら機械ガードを同じ変更で更新する。
8. 検証は自分で回し、緑を自分の目で読んでから完了と言う。サブエージェントの「緑」を鵜呑みにしない。検証と公開を同じバッチに混ぜない。
9. 小さく頻繁な PR。1 PR = 1 論理変更。指摘・依頼の範囲を勝手に広げない。
10. 秘密情報を出力に転記しない(場所だけ参照しマスク)。
11. 副作用のある検証コマンドを無確認で実行しない。
12. モード A では全エージェントを読み取り専用で動かす。
13. 不可逆操作(merge/push/deploy/公開/削除)はサブエージェントにさせず、メインが安全策の範囲で実行(破壊的/不可逆/金額/L2・ToS は人間承認)。
14. 迷ったら止まらず、安全側に倒して記録し前進する。ただし上記 1〜13 と §0 のモード選択は犠牲にしない。人間に残すべきものは needs_human に積む。

## 進め方の要約
1. §0 の 3 点(フィードバックループ条件 = 既定で「新規 finding が連続 2 round ゼロで停止 / 最大 5 round / high 以上は独立 2 名以上で多数決」、モード、範囲)を確認する。深さは別個に聞かない(深さ = ultracode、ON 前提で回す。OFF なら ON にしてもらうか『さっと 1 round』へ落とす)。回答待ちに §1 を並行して進める。
2. §1 プロファイルを作り、worktree 隔離で作業。
3. §1+§2 を全サブエージェントに渡し、実際に spawn して §3(fan-out → 敵対検証 → 統合)を回す。
4. §4 レポートを提示(= レポートのみモードの納品物)。
5. モード B かつ承認後のみ §5 で「安全・明白」だけ PR 化、判断物は needs_human に残す。
6. 各フェーズは結果(確定ドリフト + 証拠 / PR / 残課題)で締める。「やりましょうか」で止めない。不可逆操作だけ安全策に従う。
