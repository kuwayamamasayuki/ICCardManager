## 1. 言語・ランタイム

### C# (シーシャープ)

Microsoft が開発したプログラミング言語。Java に近い文法を持つオブジェクト指向言語で、Windows アプリケーション開発のデファクトスタンダード。

| 項目 | 内容 |
|------|------|
| **本プロジェクトのバージョン** | C# 10.0 |
| **なぜこのバージョンか** | Source Generator（後述）を使うために 9.0 以上が必要。かつ .NET Framework 4.8 でも使える範囲で最新の 10.0 を採用 |
| **設定箇所** | `Directory.Build.props` の `<LangVersion>10.0</LangVersion>` |

### .NET Framework 4.8

C# のコードを動かすための実行環境（ランタイム）。Windows に標準搭載されている。

| 項目 | 内容 |
|------|------|
| **なぜ .NET 8 等ではないのか** | 本システムは `felicalib.dll`（32ビット専用のネイティブDLL）に依存しており、.NET Framework の x86 ビルドが必要。また、導入先PCに追加のランタイムインストールが不要という利点もある |
| **注意点** | .NET Framework と .NET（旧 .NET Core）は別物。Stack Overflow 等で調べるときは「.NET Framework」を明記しないと、.NET 8 向けの情報がヒットして混乱する |

### .NET SDK 8.0（ビルドツールとして）

コードをコンパイル・テスト・発行するための開発ツール。アプリの実行時には使わない。

| 項目 | 内容 |
|------|------|
| **なぜランタイムと別バージョンか** | .NET SDK 8.0 は .NET Framework 4.8 向けのプロジェクトもビルドできる。CI 環境（GitHub Actions）で最新の SDK を使うことで、Source Generator 等の新機能を利用可能にしている |

## 2. UIフレームワーク（WPF）

### WPF (Windows Presentation Foundation)

Windows デスクトップアプリの UI を構築するためのフレームワーク。

| 項目 | 内容 |
|------|------|
| **読み方** | ダブリュー・ピー・エフ |
| **何ができるか** | ボタン、テキストボックス、データグリッドなどの画面部品を XAML（後述）で宣言的に配置し、C# のコードとデータバインディングで接続する |
| **Windows Forms との違い** | WPF はデータバインディングやスタイル機構が強力で、MVVM パターン（後述）と相性が良い。Windows Forms はイベントハンドラ中心 |
| **本プロジェクトでの使いどころ** | すべての画面（メイン画面、各種ダイアログ、トースト通知）が WPF で構築されている |

### XAML (ザムル)

WPF の画面レイアウトを記述するための XML ベースのマークアップ言語。

| 項目 | 内容 |
|------|------|
| **読み方** | 「ザムル」が一般的。「エックスエーエムエル」とも |
| **HTML との違い** | HTML がウェブページの構造を記述するように、XAML は WPF の画面構造を記述する。ただし XAML はデータバインディングやスタイルの仕組みが組み込まれている |
| **本プロジェクトでの使いどころ** | `Views/` フォルダ内の `.xaml` ファイル（21ファイル）。画面部品の配置、色やフォントの指定、データバインディングの定義などを行う |

### データバインディング (Data Binding)

画面上のUI部品（テキストボックスやラベル等）と、C# のプロパティを自動的に同期させる仕組み。

| 項目 | 内容 |
|------|------|
| **具体例** | XAML に `Text="{Binding StaffName}"` と書くと、ViewModel の `StaffName` プロパティの値が自動的にテキストボックスに表示される。値が変わればUIも自動更新される |
| **なぜ便利か** | UI更新のコードを手書きする必要がなくなる。「データが変わったらUIに反映」を WPF が自動でやってくれる |
| **前提条件** | バインド元のクラスが `INotifyPropertyChanged` インターフェースを実装している必要がある（CommunityToolkit.Mvvm が自動生成してくれる） |

### INotifyPropertyChanged

プロパティの値が変わったことをUIに通知するためのインターフェース。

| 項目 | 内容 |
|------|------|
| **なぜ必要か** | WPF のデータバインディングはこの通知を監視している。通知がないと、C# 側で値を変えても画面が更新されない |
| **本プロジェクトでの実装方法** | 手書きはせず、`CommunityToolkit.Mvvm` の `[ObservableProperty]` を使って自動生成している（後述） |

### ValueConverter (値コンバーター)

データバインディングの途中でデータの形式を変換する部品。

| 項目 | 内容 |
|------|------|
| **具体例** | `WarekiConverter` — 西暦の日付（2026-04-13）を和暦（令和8年4月13日）に変換して画面に表示する |
| **本プロジェクトでの使いどころ** | `Views/Converters/` フォルダ。和暦変換、カード向き名称変換など |

### 添付プロパティ (Attached Property)

WPF の部品に「後付け」で機能を追加する仕組み。

| 項目 | 内容 |
|------|------|
| **具体例** | `DataGridHighlightHelper` — 標準の DataGrid にハイライトアニメーション機能を追加している |
| **なぜ普通のプロパティではないのか** | 標準部品（DataGrid等）のソースコードは変更できないため、外部からプロパティを「貼り付ける」形で拡張する |
| **本プロジェクトでの使いどころ** | `Views/Helpers/DataGridHighlightHelper.cs` |

### ResourceDictionary (リソースディクショナリ)

色、フォント、スタイルなどの共通設定を一箇所にまとめて管理するWPFの仕組み。

| 項目 | 内容 |
|------|------|
| **CSS との類似性** | Web 開発における CSS ファイルに近い役割。見た目の定義をXAMLの各ファイルに散らさず、集約できる |
| **本プロジェクトでの使いどころ** | `Resources/Styles/AccessibilityStyles.xaml` — フォントサイズ（小/中/大/特大）やサイドバー幅の動的変更を実現 |

## 3. アーキテクチャパターン

### MVVM (Model-View-ViewModel)

画面（View）、画面のロジック（ViewModel）、データや業務ロジック（Model）を分離する設計パターン。

| 項目 | 内容 |
|------|------|
| **読み方** | エム・ブイ・ブイ・エム |
| **なぜ分離するのか** | UIのコードとビジネスロジックが混ざると、テストが困難になり、変更時の影響範囲が読みにくくなる。分離することで ViewModel だけを単体テストできる |
| **3つの役割** | **Model** = データ構造・DB操作・業務計算（`Models/`, `Services/`, `Data/`）、**View** = XAML画面（`Views/`）、**ViewModel** = ViewとModelの仲介役（`ViewModels/`） |
| **本プロジェクトでの方針** | View のコードビハインド（`.xaml.cs`）にはロジックを書かず、ViewModel にデータバインディングで接続する |

### DI (Dependency Injection / 依存性注入)

クラスが必要とする部品（依存オブジェクト）を、クラス自身が作るのではなく、外部から渡してもらう設計手法。

| 項目 | 内容 |
|------|------|
| **読み方** | ディー・アイ |
| **なぜ使うのか** | テスト時に本物の部品をモック（偽物）に差し替えられる。例: テスト時は `FelicaCardReader`（本物のカードリーダー）の代わりに `MockCardReader` を注入する |
| **具体例** | `MainViewModel` のコンストラクタが `ICardReader` を受け取る → 本番では `FelicaCardReader`、テストでは `MockCardReader` が注入される |
| **設定箇所** | `App.xaml.cs` の `ConfigureServices()` メソッド（170〜330行目付近）にすべてのサービス登録がまとまっている |
| **関連用語** | → DIコンテナ、→ サービス登録（Singleton / Transient） |

### DIコンテナ (DI Container)

DI で注入するオブジェクトの生成・管理を自動化してくれるライブラリ。

| 項目 | 内容 |
|------|------|
| **本プロジェクトで使用** | `Microsoft.Extensions.DependencyInjection` |
| **Singleton と Transient** | **Singleton** = アプリ全体で1つのインスタンスを使い回す（例: `DbContext`、各リポジトリ）。**Transient** = 要求のたびに新しいインスタンスを作る（例: ViewModel、ダイアログ） |
| **なぜ使い分けるか** | DB接続のような共有リソースは Singleton にしないと接続が増えすぎる。ViewModel は画面を開くたびに新しい状態で始めたいので Transient |

### Repository パターン (リポジトリパターン)

データベースへのアクセスを専用のクラス（リポジトリ）に集約し、他のコードからはインターフェース経由でデータ取得・保存を行う設計パターン。

| 項目 | 内容 |
|------|------|
| **なぜ使うのか** | SQL 文がアプリ全体に散らばるのを防ぐ。DB の実装を変えてもリポジトリの中だけ修正すればよい |
| **本プロジェクトのリポジトリ** | `IStaffRepository`（職員）、`ICardRepository`（カード）、`ILedgerRepository`（出納記録）、`ISettingsRepository`（設定）、`IOperationLogRepository`（操作ログ） |
| **ファイルの場所** | `Data/Repositories/` |

### Service Layer パターン (サービス層パターン)

業務ロジックを「サービスクラス」にまとめ、ViewModel やリポジトリから独立させる設計パターン。

| 項目 | 内容 |
|------|------|
| **なぜ使うのか** | 複数の ViewModel から同じ業務ロジックを呼ぶ場合にコードが重複しない。テストもしやすい |
| **本プロジェクトの主なサービス** | `LendingService`（貸出/返却）、`ReportService`（帳票）、`BackupService`（バックアップ）、`DashboardService`（ダッシュボード集計）、`CsvImportService`（CSVインポート）など20以上 |
| **ファイルの場所** | `Services/` |

### Value Object (値オブジェクト)

「値」そのものを表現する小さなクラス。等価性は内部の値で判断する（参照ではなく）。

| 項目 | 内容 |
|------|------|
| **具体例** | `Money` — 金額を表す。負の金額やオーバーフローを防ぐバリデーションを内包する。`FiscalYear` — 年度を表す。4月始まりの年度計算ロジックを内包する |
| **なぜ `int` や `string` を直接使わないのか** | 「金額」を `int` で扱うと、間違えて「個数」と足し算できてしまう。型で意味を区別することでバグを防ぐ |
| **ファイルの場所** | `Common/ValueObjects/` |

### Pub/Sub メッセージング (Publish/Subscribe)

コンポーネント同士が直接参照せずに、メッセージを介して通信する仕組み。

| 項目 | 内容 |
|------|------|
| **なぜ使うのか** | ViewModel 同士が直接参照すると依存関係が複雑になる。メッセージングなら送信側と受信側が互いを知らなくてよい |
| **本プロジェクトでの実装** | `CommunityToolkit.Mvvm.Messaging` の `WeakReferenceMessenger` を使用。例: カード読み取り抑制メッセージ（`CardReadingSuppressedMessage`）をダイアログから送信し、メイン画面が受信して読み取りを一時停止する |

## 4. MVVMフレームワーク（CommunityToolkit.Mvvm）

### CommunityToolkit.Mvvm

Microsoft 公式の軽量 MVVM フレームワーク。旧称「MVVM Toolkit」。

| 項目 | 内容 |
|------|------|
| **バージョン** | 8.2.2 |
| **なぜ使うのか** | MVVM に必要な定型コード（INotifyPropertyChanged の実装、ICommand の実装等）を大幅に削減できる |
| **特徴** | Source Generator（後述）を使い、属性を付けるだけでコードを自動生成してくれる |

### [ObservableProperty] 属性

`private` フィールドに付けると、通知機能付きの `public` プロパティを自動生成してくれる属性。

| 項目 | 内容 |
|------|------|
| **書き方** | `[ObservableProperty] private string _staffName;` → `StaffName` プロパティが自動生成される |
| **何が生成されるか** | プロパティの getter/setter、値変更時の `PropertyChanged` イベント発火、`OnStaffNameChanged()` コールバックメソッド |
| **注意点** | フィールド名は `_camelCase`（アンダースコア + 小文字始まり）で書く。生成されるプロパティ名は `PascalCase` になる |
| **本プロジェクトでの使用数** | 16以上の ViewModel で使用 |

### [RelayCommand] 属性

メソッドに付けると、`ICommand` 実装を自動生成してくれる属性。

| 項目 | 内容 |
|------|------|
| **ICommand とは** | WPF のボタン等のクリック操作を ViewModel のメソッドに結びつけるためのインターフェース |
| **書き方** | `[RelayCommand] private void Save() { ... }` → `SaveCommand` プロパティが自動生成される |
| **async 対応** | `[RelayCommand] private async Task SaveAsync()` のように非同期メソッドにも使える。実行中は自動的にボタンが無効化される |
| **XAML での使い方** | `Command="{Binding SaveCommand}"` でバインドする |

### ObservableObject

`INotifyPropertyChanged` を実装済みの基底クラス。

| 項目 | 内容 |
|------|------|
| **本プロジェクトでの使い方** | `ViewModelBase` が `ObservableObject` を継承し、すべての ViewModel が `ViewModelBase` を継承する二段構成 |
| **ViewModelBase の追加機能** | ビジー状態管理（`IsBusy`）、進捗表示（`Progress`）、キャンセルトークン（`CancellationTokenSource`） |

### Source Generator (ソースジェネレーター)

コンパイル時にC#のソースコードを自動生成する仕組み。

| 項目 | 内容 |
|------|------|
| **T4テンプレートとの違い** | T4 は手動で実行するが、Source Generator はビルドのたびに自動で動く。コードの変更に追従して常に最新の状態が生成される |
| **本プロジェクトでの使いどころ** | `[ObservableProperty]` や `[RelayCommand]` の属性を読み取り、対応するプロパティやコマンドのコードを生成する |
| **partial class が必要な理由** | 手書きのコードと自動生成コードを同じクラスに合体させるために、`partial` キーワードが必須 |

### WeakReferenceMessenger

ViewModel 同士がメッセージを送受信するための仲介役。

| 項目 | 内容 |
|------|------|
| **Weak の意味** | メッセージ購読者への参照を「弱参照」で保持するため、購読者が不要になればガベージコレクションで自動回収される。メモリリークを防ぐ |
| **本プロジェクトでの使い方** | `App.xaml.cs` で `WeakReferenceMessenger.Default` を Singleton 登録し、各 ViewModel がコンストラクタで受け取る |

## 5. 依存性注入（DI）

### Microsoft.Extensions.DependencyInjection

Microsoft 公式の DI コンテナライブラリ。

| 項目 | 内容 |
|------|------|
| **バージョン** | 8.0.1 |
| **なぜこれを選んだか** | ASP.NET Core と同じ DI コンテナなので、情報量が多く学習コストが低い。軽量で WPF にも使える |

### Microsoft.Extensions.Hosting

アプリケーションの起動・終了・設定読み込みを統一的に管理するフレームワーク。

| 項目 | 内容 |
|------|------|
| **バージョン** | 8.0.1 |
| **何をしてくれるか** | DI コンテナの構築、`appsettings.json` の読み込み、ロギングの設定などを1つの「ホスト」にまとめて管理する |

### Microsoft.Extensions.Configuration.Json

`appsettings.json`（JSON形式の設定ファイル）を読み込むためのライブラリ。

| 項目 | 内容 |
|------|------|
| **本プロジェクトでの設定項目** | キャッシュTTL、タイムアウト値、組織種別、データベースパスなど |
| **Options パターン** | 設定値を `AppOptions`、`CacheOptions` などの型付きクラスに自動マッピングする仕組み。`services.Configure<AppOptions>(config)` で登録し、コンストラクタで `IOptions<AppOptions>` として受け取る |

### Microsoft.Extensions.Logging

ログ出力の共通インターフェース。

| 項目 | 内容 |
|------|------|
| **なぜ直接ファイルに書かないのか** | `ILogger<T>` インターフェースを使うことで、出力先（コンソール、ファイル、デバッグウィンドウ）を切り替えられる。テスト時はログ出力を無効化することもできる |
| **本プロジェクトのカスタム実装** | `FileLogger` / `FileLoggerProvider`（`Infrastructure/Logging/`）— ファイルローテーション（30日保持、10MB上限）を備えた独自のファイルロガー |

### Microsoft.Extensions.Caching.Memory

インメモリキャッシュ（メモリ上に一時的にデータを保存して高速に再利用する仕組み）を提供するライブラリ。

| 項目 | 内容 |
|------|------|
| **なぜ使うのか** | 毎回 DB に問い合わせると遅い。一度取得したデータをメモリに保持し、一定時間はDB問い合わせを省略する |
| **TTL (Time To Live)** | キャッシュの有効期限。期限が切れると次回アクセス時に DB から再取得する |
| **共有モードでの調整** | スタンドアロン時は TTL 60秒、共有モード時は 10〜15秒に短縮（他PCの変更を早く反映するため） |
| **本プロジェクトでの設定** | `Infrastructure/Caching/CacheOptions.cs`、`CacheKeys.cs` |

## 6. データベース（SQLite）

### SQLite (エスキューライト)

ファイルベースの軽量リレーショナルデータベース。サーバー不要で、1つの `.db` ファイルにすべてのデータが格納される。

| 項目 | 内容 |
|------|------|
| **なぜ SQLite か** | インターネット非接続環境で動作する必要があり、SQL Server 等のサーバー型 DB は使えない。インストールも不要で、アプリと一緒に配布できる |
| **System.Data.SQLite.Core** | C# から SQLite を操作するための NuGet パッケージ（v1.0.119） |
| **注意点** | Entity Framework（ORM）は使っていない。SQL 文を直接書いてリポジトリクラス内で実行している |

### ORM (Object-Relational Mapping)

DB のテーブルと C# のクラスを自動的に対応付ける技術。Entity Framework が代表例。

| 項目 | 内容 |
|------|------|
| **本プロジェクトでは不使用** | 本プロジェクトでは ORM を使わず、SQL を直接記述している。これは監査要件に対して SQL の挙動を厳密に制御するため |
| **関連ファイル** | `Data/Repositories/` 内の各リポジトリクラスに SQL 文が記述されている |

### マイグレーション (Migration)

DB のテーブル構造（スキーマ）を段階的に変更していく仕組み。

| 項目 | 内容 |
|------|------|
| **なぜ必要か** | アプリのバージョンアップで「テーブルに列を追加したい」等の要件が発生する。しかしユーザーの既存データは消せない。マイグレーションなら既存データを保ったまま構造を変更できる |
| **本プロジェクトの実装** | Entity Framework のマイグレーションではなく、独自の `MigrationRunner` を実装。アプリ起動時に未適用のマイグレーションを自動実行する |
| **マイグレーションクラス** | `Data/Migrations/` に `Migration_001_Initial` 〜 `Migration_009_AddCarryoverTotals` の9つ |

### journal_mode (ジャーナルモード)

SQLite がトランザクション（一連のDB操作をまとめて確定する仕組み）の安全性を保証する方式。

| 項目 | 内容 |
|------|------|
| **DELETE モード** | 本プロジェクトで使用。トランザクション完了後にジャーナルファイルを削除する。ネットワーク越しでも安全に動作する |
| **WAL モード** | 一般的には高速だが、SMB 共有フォルダ上では正しく動作しないため、本プロジェクトでは使用しない |

### busy_timeout (ビジータイムアウト)

別のプロセスが DB をロック中の場合に、何ミリ秒待機してからエラーにするかの設定。

| 項目 | 内容 |
|------|------|
| **設定値** | スタンドアロン: 5,000ms、共有モード: 15,000ms |
| **なぜ共有モードで長いか** | 複数PCが同時にアクセスするため、ロック競合が頻繁に発生する。待機時間を長くして「待てばアクセスできる」状況を増やす |

### SQLite Backup API

DB ファイルを安全にコピー（バックアップ）するための SQLite 組み込み機能。

| 項目 | 内容 |
|------|------|
| **なぜ普通のファイルコピーではないのか** | SQLite は書き込み中にファイルをコピーするとデータが壊れる可能性がある。Backup API はロックを適切に管理しながらコピーするため安全 |
| **本プロジェクトでの使いどころ** | `BackupService` — アプリ起動時の自動バックアップ、手動バックアップ機能 |

### VACUUM (バキューム)

SQLite のデータファイルを最適化（不要な空き領域を詰めてファイルサイズを縮小）するコマンド。

| 項目 | 内容 |
|------|------|
| **本プロジェクトでの実行タイミング** | 月に1回、自動実行。共有モードでは失敗する場合があり、次回起動時に再試行する |

## 10. ビルド・パッケージ管理

### NuGet (ヌゲット)

.NET の公式パッケージマネージャー。外部ライブラリの取得・管理を行う。

| 項目 | 内容 |

| 項目 | 内容 |
|------|------|

| **Directory.Build.props** | ソリューション内の全プロジェクトに共通するビルド設定を一箇所にまとめるファイル。本プロジェクトでは C# バージョンやロックファイル設定を定義 |

## 11. CI/CD（継続的インテグレーション/デリバリー）

### CI (Continuous Integration / 継続的インテグレーション)

コードの変更を自動的にビルド・テストする仕組み。

| 項目 | 内容 |
|------|------|
| **なぜ必要か** | コードの変更が既存機能を壊していないか、人手に頼らず毎回自動チェックできる |
| **本プロジェクトのCI** | GitHub Actions を使用。`main` / `develop` ブランチへのプッシュ、PR 作成時に自動実行される |

### GitHub Actions

GitHub が提供する CI/CD サービス。リポジトリに YAML ファイルを置くだけで自動化パイプラインが動く。

| 項目 | 内容 |
|------|------|
| **ワークフローファイル** | `.github/workflows/ci.yml`（ビルド・テスト）、`.github/workflows/release.yml`（リリース） |
| **ci.yml の処理内容** | ビルド → テスト実行（UI テストは除外）→ コードフォーマットチェック → カバレッジ送信 |
| **release.yml の処理内容** | `v*` タグ作成で発火 → Release ビルド → ZIP 作成 → GitHub Release に公開 |

### codecov

コードカバレッジの結果を可視化・追跡するサービス。

| 項目 | 内容 |
|------|------|
| **本プロジェクトでの役割** | CI で計測したカバレッジを PR にコメントとして表示し、カバレッジの増減を追跡する |

### dotnet format

C# のコードスタイル（インデント、命名規則等）を自動チェック・修正するツール。

| 項目 | 内容 |
|------|------|
| **CI での使い方** | `dotnet format --verify-no-changes` でスタイル違反がないか確認。違反があればCIが失敗する |
| **ルール定義** | `.editorconfig` ファイル |

## 13. 開発支援ツール・設定

### .editorconfig (エディターコンフィグ)

コードの書き方ルール（インデント、命名規則、括弧の使い方など）をファイルで定義し、チーム全体で統一するための設定ファイル。

| 項目 | 内容 |
|------|------|
| **Visual Studio との連携** | Visual Studio が自動的にこのファイルを読み込み、ルールに従ったフォーマットや警告を提示する |
| **本プロジェクトの主なルール** | private フィールドは `_camelCase`、`var` の使用を推奨、パターンマッチング推奨、インデントは4スペース（XML/XAML/JSON は2スペース） |

### appsettings.json

アプリの設定値を外部ファイルに定義するための JSON ファイル。

| 項目 | 内容 |
|------|------|
| **なぜコード内に直書きしないのか** | 設定値をコードに埋め込むと、値を変えるたびにビルドし直す必要がある。外部ファイルにすれば再ビルド不要で変更できる |
| **本プロジェクトの設定項目** | キャッシュTTL、タイムアウト値、組織種別、ログレベルなど |

### IDisposable (アイディスポーザブル)

使い終わったリソース（ファイルハンドル、DB接続など）を明示的に解放するためのインターフェース。

| 項目 | 内容 |
|------|------|
| **なぜ必要か** | ガベージコレクションはメモリは自動回収するが、ファイルや DB 接続などの「アンマネージドリソース」は自動解放されない。`Dispose()` で明示的に解放する |
| **using 文** | `using var conn = new SQLiteConnection(...)` と書くと、スコープを抜けるときに自動的に `Dispose()` が呼ばれる |

### async/await (エイシンク/アウェイト)

時間のかかる処理（DB問い合わせ、ファイル読み書きなど）を、UIをフリーズさせずに実行するための非同期プログラミング機構。

| 項目 | 内容 |
|------|------|
| **なぜ重要か** | WPF は単一の UI スレッドで画面を描画している。ここで長い処理を同期実行すると画面がフリーズする。`async/await` で別スレッドに処理を逃がし、完了後に UI スレッドに戻る |
| **本プロジェクトでの方針** | サービスのメソッドには `Async` サフィックスを付ける（例: `SaveAsync`）。ViewModel の `[RelayCommand]` も async 対応 |

### CancellationToken (キャンセレーショントークン)

実行中の非同期処理をキャンセルするための仕組み。

| 項目 | 内容 |
|------|------|
| **本プロジェクトでの使い方** | `ViewModelBase` が `CancellationTokenSource` を保持し、画面遷移や明示的なキャンセル操作で処理を中断できる |

### DebugDataViewer

開発時にカードの生データや DB の状態を確認するためのデバッグ専用ツール（別アプリ）。

| 項目 | 内容 |
|------|------|
| **ファイルの場所** | `tools/DebugDataViewer/` |
| **用途** | カードリーダーで読み取った生のバイト列の確認、DB テーブルの中身の閲覧など |
