# SummaryGenerator テスト結果レポート

**実施日**: 2024年12月10日（更新）

**テストファイル**: `tests/ICCardManager.Tests/Services/SummaryGeneratorComprehensiveTests.cs`

---

## 概要

### 新仕様の要件

1. **日付ごとに分割**: 履歴は日付ごとに分割して物品出納簿に記載
2. **利用とチャージを別行に**: 同じ日でも鉄道・バスの利用とチャージは別の行
3. **古い順にソート**: ICカード履歴は新しい順だが、物品出納簿は古い順に出力
4. **同日の時刻順**: 同日に利用とチャージがある場合、時刻順（古い順）に出力
5. **往復表示の基準**: 往復は古い方の行程を基準に表示

### テスト対象メソッド

- `GenerateByDate`: 日付ごとに分割し、利用とチャージを別行にして返す（新仕様）
- `Generate`: 従来のメソッド（互換性維持）

## 重要な仕様変更（2024年12月10日更新）

### 1. 同日の利用とチャージの順序

ICカード履歴は新しい順で格納されているため、インデックスが大きいほど古いエントリとなります。
同日に利用とチャージがある場合、**古い方から順に**出力します。

例: TC004
- 入力: [0]利用(新しい), [1]チャージ(古い)
- 出力: [0]チャージ, [1]利用 （古い順）

### 2. 往復表示の基準

往復は**古い方の行程**を基準に表示します。

例: TC008
- 入力: [0]博多→天神(帰り・新しい), [1]天神→博多(行き・古い)
- 出力: 「天神～博多 往復」（古い方=行きを基準に）

### 3. 乗継の判定

乗継は**古い順**に処理して判定します。降車駅と次の乗車駅が同じ場合に統合されます。

例: TC010
- 入力: [0]中洲川端→貝塚(新しい), [1]天神→中洲川端(古い)
- 処理順序: 天神→中洲川端 → 中洲川端→貝塚
- 出力: 「天神～貝塚」（乗継として統合）

### 4. 循環移動の表示（2024年12月11日追加）

乗継統合の結果、起点と終点が同じになる場合（循環移動）は、統合せずに**個別区間を表示**します。
「A～A」という表示は移動がないように見えてしまうため、実際の移動経路を明示します。

例: TC014（12/8）
- 入力: [5]天神→姪浜, [4]姪浜→西新, [3]西新→天神
- 統合結果: 起点=天神, 終点=天神（循環）
- 出力: 「天神～姪浜、姪浜～西新、西新～天神」（個別区間で表示）

**注意**: TC020（同一駅乗降）のように単一の「天神→天神」という記録は、そもそも循環移動ではなく改札を通過しただけなので「天神～天神」のまま表示されます。
